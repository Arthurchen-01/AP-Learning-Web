# 聊天框控制器方案

更新时间：2026-03-28 00:18
目标：让 OpenClaw 对话更容易切新话题，并控制聊天框/旧会话数量，减少分支堆积。

---

## 一、目标行为

需要实现的行为：
1. 用户随时可以切换到新的话题
2. 聊天框总数尽量保持在 **4 个及以内**
3. 每删除一个聊天框/旧会话，就补一个新的空白可用框
4. 新框需要带备注：`更改时间：YYYY-MM-DD HH:mm`
5. 每累计 **3 次切换 / 新建**：
   - 检查 `C:\Users\25472\Desktop\记忆\当日记忆.md`
   - 如果文件已更新：清理其它旧聊天框 / 旧会话 / 多余分支
   - 如果没更新：先不清，避免上下文还没沉淀就丢掉

注意：
- 这里的“删掉其余内容”明确指：**删掉其它聊天框 / 旧会话 / 多余分支**
- **不删除记忆文件正文**

## 二、当前已经完成的基础工作

### 已完成
- 已把 `agents.defaults.subagents.maxConcurrent` 从 8 降到 2
- 已开启更积极的 `session.maintenance`
- 这能先缓解分支泛滥，但还不能实现“固定 4 框 + 自动补位”

### 结论
OpenClaw 配置层可以解决：
- subagent 太多
- 会话堆积太快

但不能单独解决：
- 聊天框数量控制
- 删除一个自动补一个
- 每 3 次切换检查当日记忆

因此需要额外做一个 **聊天框控制器逻辑**。

## 三、推荐实现方式

## 方案 A（推荐）：外层脚本 / 小管理器

做一个轻量控制器，负责：
- 记录当前活跃聊天框/会话
- 记录每次“新建/切换”的计数
- 检查 `当日记忆.md` 是否在最近有变化
- 超过阈值后自动清理旧会话并补新框

### 这个控制器要维护的状态
建议用一个状态文件，例如：
- `C:\Users\25472\Desktop\记忆\chatbox-state.json`

示例：
```json
{
  "switchCount": 2,
  "lastMemoryMtime": "2026-03-28T00:10:00+08:00",
  "boxes": [
    { "id": "box-1", "createdAt": "2026-03-28 00:01", "note": "更改时间：2026-03-28 00:01" },
    { "id": "box-2", "createdAt": "2026-03-28 00:05", "note": "更改时间：2026-03-28 00:05" }
  ]
}
```

### 控制器的核心逻辑
1. 新建或切换话题时：
   - `switchCount += 1`
2. 如果聊天框数量 > 4：
   - 优先删最老/最不活跃的旧会话
3. 如果有框被删：
   - 自动补一个新框
   - 备注写入当前时间
4. 如果 `switchCount >= 3`：
   - 检查 `当日记忆.md` 修改时间
   - 若比上次记录更新：
     - 清理旧聊天框/旧会话/多余分支
     - `switchCount = 0`
     - 更新 `lastMemoryMtime`
   - 若未更新：
     - 只提醒，不清理

## 四、哪些部分可以直接自动化，哪些不能

## 能直接自动化的
- 记录切换次数
- 检查 `当日记忆.md` 的修改时间
- 管理一个本地状态文件
- 根据规则输出“应该保留哪些会话/应该清掉哪些旧会话”的决策

## 不一定能直接自动化的
- OpenClaw control UI 里“聊天框”的真实创建/删除动作，如果没有现成 API，就不能凭空点 UI
- 如果 control UI 不暴露创建新聊天框/重命名/备注接口，就需要半自动：
  - 先生成建议清单
  - 再由用户一键执行或手动点一下

### 第 1 阶段（已做一部分）
- 收紧 subagent

### 第 2 阶段（建议现在做）

- 先生成待执行动作清单
- 由用户一键或手动触发
