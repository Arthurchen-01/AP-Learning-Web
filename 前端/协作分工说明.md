# Chrome 插件协作分工说明

更新时间：2026-03-27 20:46
目标：并行推进一个“进入 AP 模考即自动采集并下载关键数据”的 Chrome/Edge 扩展。

---

## 一、项目目标

要做的不是普通抓包工具，而是一个 **AP 模考专项采集插件**。

目标行为：
- 当用户进入 AP 模考页面时，插件自动开始记录。
- 在用户进行以下关键动作时，自动采集并下载数据：
  - start / resume
  - preparing page
  - question page
  - next / previous
  - flag
  - review page
  - module 切换
  - directions / timer / more / calculator / line reader / notes / highlights 等工具交互
- 自动导出以下信息：
  - request payload
  - response payload
  - localStorage / sessionStorage / IndexedDB 快照与 diff
  - URL 参数变化
  - 页面注水状态（如 `__NEXT_DATA__`、`__INITIAL_STATE__`）
  - DOM / 工具栏可见性规则
  - 关键 CSS / 静态资源索引

最终希望浏览器自动下载到指定下载目录下的一个结构化 session 文件夹里。

---

## 二、协作方式

采用 **双 AI 并行**：
- 我（当前 AI）负责：**先搭插件骨架 + 跑通最小可用版本（MVP）**
- 另一个 AI 负责：**补充采集能力设计、输出格式设计、规则层设计、以及可选高级功能方案**

这样能避免两边重复写同一套基础代码。

---

## 三、我（当前 AI）负责的部分

### 目标
优先交付一个可以尽快开始跑的 Chrome 扩展骨架，先把“能抓到”这件事做出来。

### 我负责的任务范围

#### 1. 插件项目骨架
- 建立 Chrome/Edge 扩展目录结构
- 生成：
  - `manifest.json`
  - `background/service worker`
  - `content script`
  - `options page`（基础版即可）
  - 可选：popup 或 devtools 面板骨架

#### 2. MVP 网络采集能力
先实现最重要的：
- 拦截 `fetch`
- 拦截 `XMLHttpRequest`
- 记录：
  - request URL
  - method
  - headers
  - request body
  - response status
  - response body（可读时）
  - timestamp
- 在关键事件时输出 JSON 文件下载

#### 3. MVP 页面状态采集
- 采集 `location.href`
- 采集 `localStorage`
- 采集 `sessionStorage`
- 采集页面注水状态：
  - `__NEXT_DATA__`
  - `__INITIAL_STATE__`
  - `window.__NUXT__`
  - 其他常见注水对象（如果存在）

#### 4. MVP 事件机制
- 自动识别 AP 模考页面是否开始
- 关键交互时打点并记录 event：
  - start
  - resume
  - next
  - previous
  - flag
  - review
  - module change
- 每个 event 都触发一次快照导出

#### 5. 下载落盘策略（浏览器可行版本）
- 设计基于 `chrome.downloads.download` 的输出方式
- 先输出到浏览器默认下载目录下的子路径
- 文件命名要可追踪：
  - `session-manifest.json`
  - `event-xxx.json`
  - `network-xxx.json`
  - `storage-xxx.json`

### 我不优先负责的内容
这些我先不主做，留给另一个 AI 或第二阶段：
- IndexedDB 深层完整导出
- CSS computed style 的精细抓取
- 复杂 DOM 组件语义抽取
- 工具栏业务规则的最终结构化模型
- 高级打包器 / 自动 zip / HAR 全量标准化

## 四、另一个 AI 负责的部分

### 目标
在我搭建 MVP 骨架时，另一个 AI 重点负责 **高层设计、数据规范、规则抽象、增强能力方案**，保证后续不用返工。

### 另一个 AI 负责的任务范围

#### 1. 数据输出规范设计
请设计一套稳定的输出结构，至少包括：
- `manifest.json`
- `timeline.json`
- `requests/*.json`
- `responses/*.json`
- `storage/*.json`
- `dom/*.html`
- `injected-state/*.json`
- `rules/*.json`
- `resources/*.json`

要求说明：
- 每类文件记录什么
- 字段怎么命名
- 哪些字段是必须的
- 哪些字段是可选的
- 如何保证后续脚本能二次处理这些数据

#### 2. 工具栏 / 交互规则模型
请整理出一套“考试工具规则配置”的建议结构，例如：
- 哪些工具在哪个 section / part 出现
- calculator 何时出现
- notes / highlight / more / line reader 的开关规则
- directions 是否可重复打开
- hide/show timer 的状态模型

请输出一个建议配置格式，例如：
```json
{
  "section_i_part_a": {
    "showCalculator": false,
    "showNotes": true,
    "showHighlights": true,
    "showMore": true
  }
}
```

#### 3. IndexedDB / storage 深度采集方案
请重点研究并给出：
- 如何在扩展里更稳定地导出 IndexedDB
- 如何做快照与 diff
- 如何区分 exam state / answer cache / timer cache / progress cache
- 哪种导出格式最适合后续分析

#### 4. DOM / 渲染资源采集方案
请整理：
- 关键页面组件应该采哪些 DOM 快照
- CSS / className / 资源 URL 应如何记录
- 数学公式（MathML / KaTeX / 原生 HTML）应该如何分类标记
- 截图、HTML、computed style 三者如何配合

#### 5. HAR / 会话归档增强方案
请评估是否应该加入：
- HAR 文件导出
- console 日志导出
- cookie 导出
- screenshots
- session zip 打包

并写明：
- 哪些值得做
- 哪些可以后做
- 哪些浏览器权限成本高

#### 6. GitHub / 开源参考调研补充
请继续搜现成开源项目，并告诉我们：
- 哪些仓库最值得借鉴
- 哪些可以直接抄结构
- 哪些只适合参考思路
- 哪些仓库过旧/不推荐

## 五、接口约定（避免两边打架）

### 我这边先默认的最小输出模型
另一个 AI 在设计时，尽量兼容下面这些顶层对象：

```json
{
  "sessionId": "...",
  "capturedAt": "...",
  "url": "...",
  "eventType": "...",
  "network": [],
  "storage": {},
  "injectedState": {},
  "dom": {},
  "notes": []
}
```

### 命名原则
- 文件名尽量英文、稳定、可排序
- 字段名统一 camelCase 或 snake_case，不要混用（建议 camelCase）
- 时间统一 ISO 8601
- 所有 event 要有时间戳和页面 URL
